# OCIE — Guideline ↔ Pipeline Mapping (Next.js + Postgres)

NSCLC drug-to-guideline mapping by biomarker and line of therapy. For every biomarker, see which drugs are current standard-of-care at each guideline line (NCCN/ASCO), with pipeline drugs overlaid on the same grid, NLP-tagged with evidence-trial mentions pulled from the guideline text itself.

This is the Next.js + Postgres rebuild of the earlier static-HTML prototype, structured for a real deploy on Vercel with a cloud Postgres backend instead of a flat JSON file. See `PROJECT_BRIEF.md` for the original scope doc.

**Status: fully verified working.** Type-checked (`tsc --noEmit`), linted (`eslint`), production-built (`next build`), and run end-to-end against a real local Postgres instance — schema applied, seeded (38 biomarkers / 98 drugs / 186 occurrences), dev server started, and the rendered page confirmed showing live DB data, not placeholder content.

---

## Stack

- **Next.js 16** (App Router, Server Components) + **TypeScript** + **Tailwind CSS v4**
- **Postgres** — host-agnostic via `DATABASE_URL` (Supabase, Neon, or Vercel Postgres all work unchanged)
- **No ORM** — plain `pg` with typed query functions in `src/lib/queries.ts`. Small, fixed schema; an ORM would add a layer without adding safety here.
- Data pipeline: **Python** (`scripts/parse_guidelines.py` + `scripts/nlp_utils.py`) parses the source xlsx and runs the NLP enrichment layer → JSON → **Node seed script** (`scripts/seed.ts`) loads it into Postgres.

## Data model

Three tables, mirroring `scripts/parse_guidelines.py`'s output exactly:

- **`biomarkers`** — one row per biomarker/mutation/stage-combination, tagged by track (A/B/C/Driver-Negative/Uncommon), with `notable_trials` (NLP-extracted from guideline notes).
- **`drugs`** — one row per distinct drug/regimen, `source` is `guideline` or `missing_drugs`.
- **`occurrences`** — the actual mapping: `(drug, biomarker, line)` plus `status` (`current_soc` / `pipeline_pending` / `ambiguous`), `status_detail` (`white_space_gap` / `pipeline_signal` / `unclear`, NLP-derived for ambiguous rows), and `evidence_trials` (NLP-extracted trial names from the source text).

The 3-state `status` plus 3-state `status_detail` collapse into a single 5-state `VisualStatus` (`src/types/index.ts: visualStatus()`) that drives every status color in the UI.

## Local setup

```bash
npm install

# 1. Database — point DATABASE_URL at Supabase, Neon, Vercel Postgres, or local Postgres
cp .env.example .env.local
# edit .env.local with your connection string

# 2. Apply schema (paste db/schema.sql into the Supabase SQL editor, or:)
psql "$DATABASE_URL" -f db/schema.sql

# 3. Seed (data/guideline_mapping.json is already generated and committed —
#    you only need to re-run the Python parser if you change the source xlsx)
npm run db:seed

# 4. Run
npm run dev
```

Visit `http://localhost:3000`. If `DATABASE_URL` isn't set or the DB is empty, the page shows a setup screen with these same steps instead of erroring — it won't silently break if you open it before step 1–3 are done.

**To re-run the ETL after editing the source xlsx:**
```bash
pip install -r scripts/requirements.txt
npm run etl        # re-parses xlsx -> data/guideline_mapping.json
npm run db:seed    # re-seeds Postgres from the refreshed JSON
```

## Deploying

**Database (pick one — both work with zero code changes, same `DATABASE_URL`):**
- *Supabase*: new project → Settings → Database → connection string (use the **Transaction pooler** URI for serverless compatibility) → run `db/schema.sql` in the SQL editor → `npm run db:seed` locally pointed at it.
- *Neon / Vercel Postgres*: create from the Vercel dashboard (Storage tab) — connection string is provided directly, same steps otherwise.

**App (Vercel):**
1. Push this repo to GitHub.
2. Import into Vercel.
3. Add the `DATABASE_URL` environment variable in Vercel project settings (same value as `.env.local`).
4. Deploy. No other config needed — `next build` is already verified clean.

## Design decisions worth knowing about

- **Status colors are a 5-state present-state classification, not the reference mockup's 4-state timeline.** The mockup's white/blue/green/peach scheme encodes *when* a drug is projected to enter guidelines (now/3yr/5yr/gap) — we don't have projection data, so our colors encode *current confidence* instead (SOC / pending / hedged-with-signal / explicit-gap / unclear). Visually similar pattern, different underlying meaning — see `STATUS_LABEL` in `src/types/index.ts`.
- **Candidates and Scenarios tabs are present but disabled** (`src/components/TabBar.tsx`) rather than omitted, so the nav matches the reference mockup's structure. They need trial-probability and time-horizon-projection data this dataset doesn't have — enabling them with fabricated numbers would be worse than greying them out.
- **Evidence-trial counts in the left rail are NOT live trial volume.** They're a count of distinct trial names the NLP layer found mentioned in that biomarker's guideline notes — labeled "mentions," not "trials," on purpose.
- **Why Postgres over Mongo/Neo4j**: the query pattern here is filter/join/group-by on a small, stable, normalized schema — exactly Postgres's strength, and it deploys natively alongside Vercel. Neo4j is the right call later if a genuinely graph-shaped feature shows up (e.g. "explore everything this drug touches"), the same way it's used for the connection-suggestion engine elsewhere in this codebase — not needed for this dashboard's current query pattern.

## Known gaps / next steps

- Biomarker fuzzy-matching (`best_biomarker_match` in `parse_guidelines.py`) is token-overlap based, not embeddings — good enough for this dataset, not perfect. Rows that land in `"Pipeline (unmatched)"` need a manual look.
- Track A/B (early-stage) has no pipeline overlay yet — the `Missing drugs` sheet doesn't cover early-stage candidates.
- NCCN/ASCO aren't version-split — the source sheets merge them into one recommendation per row.
- Trial-volume counts, P(approval) scoring, and the scenario/horizon-projection model are still out of scope — see `PROJECT_BRIEF.md`.
