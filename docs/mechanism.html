1. Overview

OCIE is a decision-support system that maps oncology treatment guidelines (NCCN, ASCO) against clinical trial data to:

  • Visualize the current Standard of Care (SOC) by biomarker and line of therapy
  • Project when pipeline drugs in clinical trials will reach SOC
  • Identify white-space gaps where unmet need exists despite ongoing trials
Core Insight: FDA approval does not equal guideline adoption. A drug must be included in NCCN/ASCO guidelines before it becomes standard of care. The timeline model accounts for this lag.

4.2 Preset Pathways

The dashboard provides two presets corresponding to the FDA approval pathway:

  • Standard — Phase 3 pivotal trial, full FDA approval, standard PDUFA review (10 mo target). Default: 2 + 8 + 5 = +15 mo
  • Accelerated — Phase 2 pivotal trial, accelerated/surrogate-endpoint approval, priority review (6 mo target). Default: 2 + 4 + 5 = +11 mo

Users can also set custom weights per component. All changes reflect immediately in the projected dates.

4.3 Monte Carlo Simulation (Probabilistic Extension)

The point-estimate formula gives a single deterministic date. In practice, each weight has inherent uncertainty — submission prep can slip, FDA reviews can go beyond PDUFA targets, NCCN panels may be delayed. Monte Carlo simulation addresses this by treating each weight as a probability distribution rather than a fixed number.

How it works:

  1. Each weight (Submission, FDA Review, NCCN Lag) is modeled as a triangular distribution — defined by a minimum, mode (most likely), and maximum value. For example, "FDA Review" for standard approval: min 6 mo, mode 8 mo (user's weight), max 14 mo.
  2. The simulation runs N = 10,000 iterations. In each iteration, it randomly samples a value from each distribution using the inverse-CDF method.
  3. Each iteration computes a total offset (sum of sampled weights) and a projected SOC date.
  4. After all iterations, the results are sorted to produce a cumulative distribution of SOC dates.
  5. Key percentiles are reported: P10 (10% chance of SOC by this date = optimistic), P50 (median = most likely), P90 (90% chance of SOC by this date = conservative).

Triangular distribution parameters (default):

WeightStandardAccelerated
SubmissionT(1, 2, 4)T(1, 2, 3)
FDA ReviewT(6, 8, 14)T(3, 4, 8)
NCCN LagT(2, 5, 10)T(2, 5, 8)

Why triangular distributions? They require only three intuitive inputs (min, mode, max) that map directly to expert judgment ("best case", "most likely", "worst case"). The mode matches the user's weight value, and the range (±50% of mode) reflects observed variance from the 6-drug validation study. Users comfortable with distributions can override the spread parameters.

Output interpretation: Instead of "SOC by 2028-Q3", the model says "There is a 50% probability of SOC by 2028-Q3 (P50), with an 80% confidence window from 2028-Q1 (P10) to 2029-Q2 (P90)." This enables risk-aware decision making — a drug with a wide P10-P90 range has higher timeline uncertainty than one with a narrow range.

4.4 Visual Representation on Dashboard

Monte Carlo sits on top of the deterministic date — it does not replace it. The deterministic P50 date remains the primary number displayed in the "Projected SOC" column. A confidence bar is added alongside it:

                     2028-09

Or as a compact badge in a table cell:

  | 2028-09-01 | ████░░░░ 80% | 2028-04 – 2029-02 |
  |  projected   confidence     P10       P90     |
  |  SOC (P50)   bar                               |

Key design decisions:

  • The deterministic date is unchanged — it is the P50 (median) of the simulation, identical to the current formula result. This means existing users see the same number as before.
  • The confidence bar is additive — it appears as a visual indicator next to the date. A narrow bar (e.g., ±3mo) signals high confidence; a wide bar (e.g., ±9mo) signals uncertainty.
  • No extra user input required — the triangular spread is derived automatically from the weight values (min = weight × 0.5, max = weight × 1.5). Advanced users can override the spread if desired.
  • Performance — 10,000 iterations run in ~2ms client-side, imperceptible to the user.
Validation fit: The ±4 mo error observed in the 6-drug study maps well to triangular spreads. For example, the Pembrolizumab outlier (late by 4.5 mo) falls at roughly P85 of the standard review distribution — an expected tail event. Monte Carlo captures this without requiring different weights for each drug.

5.3 Accuracy Assessment

The model achieves ±4 mo accuracy across the validation set. Sources of variance:

  • Pembrolizumab (late by 4.5 mo) — Breakthrough Therapy designation accelerated real-world review beyond the standard default weight.
  • Selpercatinib (early by 4.7 mo) — Submission delays after data cutoff extended the PCD→FDA gap beyond typical accelerated review.
Conclusion: At a year-scale projection, ±4 mo error is operationally acceptable. The variance reflects real-world factors (breakthrough designations, submission strategy, FDA backlog) that no fixed-weight model can fully capture — which is precisely why weights are user-configurable. Individual expert users can adjust based on their domain knowledge of specific drugs, sponsors, or regulatory timelines.

7. Biomarker Pipeline Fetcher

The fetch-biomarker-pipeline.ts script performs live ClinicalTrials.gov searches for a given biomarker:

npx tsx scripts/fetch-biomarker-pipeline.ts --biomarker="KRAS G12C"

It:

  1. Queries ClinicalTrials.gov API for active NSCLC trials matching the biomarker
  2. Extracts drug names from trial intervention arms
  3. Cross-references each drug against the SOC database (Supabase)
  4. Projects timelines using the validated formula for pipeline drugs
  5. Outputs a table (stdout) or JSON file with all results

Only two external dependencies are needed:

  • SUPABASE_URL — project endpoint
  • SUPABASE_SERVICE_ROLE_KEY — server-side API key (from Supabase Settings → API)

9. User Configuration Guide

9.1 Timeline Weights

In the Pipeline tab, three numerical inputs control the projection:

  • Submission (0–6 mo) — Time from PCD to regulatory submission
  • FDA Review (0–24 mo) — FDA review clock duration
  • NCCN Lag (0–12 mo) — Guideline adoption delay after approval

The Standard and Accelerated preset buttons restore defaults validated against the 6-drug study. Custom values persist as long as the page is loaded.

9.2 Global Filters

Sidebar filters (Biomarker, Regimen Type, Histology, Line of Therapy) apply across all four tabs simultaneously.

9.3 Running the Pipeline Fetcher

To refresh pipeline data for a specific biomarker:

npx tsx scripts/fetch-biomarker-pipeline.ts --biomarker="EGFR" --output=json

Outputs to data/pipeline_<biomarker>.json.

9.4 Refreshing SOC Data

To re-seed from the xlsx source (after schema changes or data updates):

npx tsx scripts/apply-schema.ts
npx tsx scripts/seed.ts

10. Limitations & Future Work

10.1 Known Limitations

  • PCD accuracy for accelerated drugs — ClinicalTrials.gov shows updated long-term PCDs, not the original analysis cutoff. The fetcher script handles this, but the pipeline_drugs view currently uses raw PCD from the API.
  • NCCN guideline dates — Approximated from version history. Precise publication dates would improve accuracy.
  • Inclusion/exclusion criteria — Schema exists but data not yet populated. Needed for patient-level matching.
  • Concurrent regulatory pathways — Drugs may have both accelerated and standard pathways (e.g., osimertinib got accelerated first, then full approval). The model uses a single pathway.

10.2 Planned Enhancements

  • Biomarker-specific pipeline fetcher — Live ClinicalTrials.gov querying for any biomarker (script completed, ready for dashboard integration)
  • Automated weight calibration — Bayesian update of weights as more drugs pass through the pipeline
  • Patient-level trial matching — Map inclusion/exclusion criteria from trial protocols to patient subtypes
  • Competitive pressure scores — AI-generated threat assessment per treatment cell
